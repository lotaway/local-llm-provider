十、完整系统定位

这个自动评估系统的作用是：

批量筛选模型版本

自动拦截失败样本

减少人工听音成本

提供可量化指标

它不是替代人工听感，而是前置过滤器。


上线监控期必须评估的内容清单，按优先级分层，不讨论训练阶段。

一、核心目标（上线后要防什么）

上线后你真正要监控的是：

质量是否退化

模型是否漂移

生成是否异常

是否出现大规模错误样本

二、生产级必须监控的四大指标
1️⃣ WER（可懂度）

使用
Whisper

监控方式

线上抽样 5%~10% 请求

自动计算 WER

统计滚动均值

报警阈值

日均 WER > 6% → 警告

日均 WER > 8% → 报警

单样本 WER > 20% → 直接记录异常

监控的是趋势，而不是单条数据。

2️⃣ DNSMOS（音质）

来源：Microsoft

监控方式

每日抽样统计均值

统计低于阈值比例

报警规则

平均 < 4.0 → 警告

平均 < 3.8 → 报警

单样本 < 3.5 → 标记异常

3️⃣ Speaker Similarity（如果是克隆产品）

使用
ECAPA-TDNN

监控方式

对固定参考音频做相似度

统计分布变化

报警规则

平均 < 0.80 → 警告

平均 < 0.78 → 报警

4️⃣ 生成失败率

这个经常被忽略，但非常关键。

监控：

TTS 生成报错率

生成音频为空比例

音频时长异常比例

阈值：

失败率 > 1% → 警告

3% → 报警

三、强烈建议增加的监控项
5️⃣ 音频时长异常

监控：

生成时长 / 文本长度

异常情况：

时长过短 → 丢字

时长过长 → 重复

建议：

建立“字符数 vs 时长”统计模型

6️⃣ 重复片段检测

有些模型会：

重复某段词

卡住循环

可以：

用文本去重比率

或音频相似度滑窗检测

7️⃣ 用户反馈分布

监控：

差评比例

投诉比例

重新生成次数

这个往往比自动指标更敏感。

四、趋势监控比单点更重要

重点看：

7天滑动平均

P90 / P95

异常比例

不要只看均值。

五、完整线上监控结构

六、生产级推荐报警基线

普通TTS产品：

WER < 6%

DNSMOS ≥ 4.0

失败率 < 1%

语音克隆产品：

WER < 5%

DNSMOS ≥ 4.1

相似度 ≥ 0.82

失败率 < 1%

七、一个非常关键但很多人忽略的点

上线后最大的风险不是：

“单条音频不自然”

而是：

某次模型更新后整体质量下降 0.3 分
但没有人发现

所以一定要：

固定一套测试句集

每天自动跑回归测试

版本对比曲线

八、总结

上线监控核心是：

可懂度

音质

一致性

失败率

趋势变化

不是再做模型评估，
而是做质量稳定性监控。
